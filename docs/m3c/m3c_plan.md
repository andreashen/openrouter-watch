# M3C 实施规划 — test 分支 SIT 发布与数据更新自动化评估

**目标：** 建立 `test` 分支可自动访问的 SIT 页面，并给出“数据更新迁移到 GitHub 自动化”的落地方案。  
**架构：** 延续现有 Pages 发布方式，新增 SIT 发布链路 + 数据刷新工作流，二者通过分支和触发规则解耦。  
**技术栈：** GitHub Actions、GitHub Pages、Python 3.12、Node.js 22、现有 `scripts/*.py`。

---

## 实施原则

1. **最小改造**：优先复用现有 `deploy-pages.yml` 的构建逻辑，避免重复维护。
2. **环境隔离**：SIT 与现网发布目标隔离，确保验证流量不干扰生产访问。
3. **可审计**：数据刷新结果可追踪（日志、提交记录、触发来源）。

---

## 实施阶段

## P1 — SIT 发布策略与 URL 规则定稿

- 明确 `test` 分支触发条件与发布目标。
- 确认 SIT URL 命名采用 `repo/sit` 子路径。
- 定义与主站隔离的发布规则（仓库/路径/环境）。

## P2 — SIT Pages 工作流配置

- 新增或扩展 workflow，使 `test` 分支触发 SIT 构建与发布。
- 配置 Astro 构建参数（`ASTRO_SITE`、`ASTRO_BASE`）匹配 SIT URL。
- 输出可直接访问的页面地址，供提测验收。

## P3 — 数据刷新自动化设计（评估结论落地）

- 新增 `data-refresh` 工作流，执行 `fetch -> normalize -> derive`。
- 设计触发方式：`workflow_dispatch + schedule`（必要时补 `push`）。
- 定义产物回写策略（提交哪些文件、写回哪个分支）。

## P4 — 稳定性与安全策略

- 使用 GitHub Secrets 管理 `OPENROUTER_API_KEY`（可选但推荐）。
- 增加失败处理与重试建议，记录关键日志。
- 防止自动提交触发循环（路径过滤/提交信息约定/条件判断）。

## P5 — 验证与交付

- 验证 `test` 分支提交可产出 SIT 页面且链接可访问。
- 验证数据刷新工作流可独立运行并更新目标产物。
- 更新 `docs/m3c/*` 与目录说明，形成可执行任务清单。

---

## 风险与应对

1. **风险：SIT URL 子路径规则与 GitHub Pages 物理路径不一致**
   - 应对：先确定“语义后缀”与“物理发布目标”的映射关系，再固化 workflow 变量。
2. **风险：数据刷新耗时过长导致 workflow 不稳定**
   - 应对：先采用串行稳定方案，后续再评估并发与缓存优化。
3. **风险：自动提交触发级联工作流**
   - 应对：配置路径过滤、提交标识与触发条件，避免自触发循环。
